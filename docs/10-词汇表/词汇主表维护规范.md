# 词汇主表一致性保障机制

## 一、机制概述

本机制旨在保障"旋量-太极"模型词汇表在整个项目生命周期内的准确性、完整性和一致性。通过标准化流程、规范校验、版本控制和定期审核，预防和解决词汇定义不一致、术语使用混乱、版本管理缺失等问题。

---

## 二、文件结构

```
docs/10-词汇表/
 旋量太极词汇表.md          # 主表（当前生效版本）
 定义变更日志.md             # 变更记录
 词汇关系图.md               # 词与词之间的网状关系
 词汇主表维护规范.md         # 本文件（维护规则）
```

---

## 三、词汇条目标准模板

每个定义条目统一采用以下格式：

```markdown
**词条名**：[一句定义] ||
- 模型定位：[基础操作/生命状态/关系/时间/存在主体/心灵操作/经典读解]
- 核心锚点：[与之直接对应的旋量-太极操作（平方/开方/720）]
- 校准历史：[V1V2 的变化摘要]（可选）
- 关联词：[ 词条A  词条B  ]
- 来源文章：[《文章标题》]
- 审查者：[姓名/角色]
```

### 状态标签说明

| 状态 | 标签 | 含义 | 修订难度 |
|------|------|------|----------|
| 锁定 |  | 经过多人审查且跨文章验证，不易再动 | 需多人一致同意 |
| 试用 |  | 新近读解，逻辑自洽但尚未跨文章验证 | 作者可改 |
| 待定 |  | 提出但尚未完全锚定 | 随时可动 |

## 四、标准化操作流程

### 4.1 词汇新增流程

提议  起草定义  同行审查  状态标记  录入主表  更新日志  更新关系图

#### 步骤说明：
1. **提议**：任何人可提出新增词汇建议
2. **起草定义**：按标准模板撰写
3. **同行审查**：至少1人审查通过
4. **状态标记**：初始标记为「试用」
5. **录入主表**：按分类插入相应章节
6. **更新日志**：记录新增信息
7. **更新关系图**：添加关联关系

### 4.2 词汇修改流程

发现问题  评估影响  修改定义  审查确认  更新主表  更新日志  更新关系图

#### 步骤说明：
1. **发现问题**：通过触发器发现不一致
2. **评估影响**：分析关联词汇和引用文档
3. **修改定义**：保持核心锚点一致性
4. **审查确认**：根据状态标签确定审查级别
5. **更新主表**：修改条目内容
6. **更新日志**：记录修改详情和原因
7. **更新关系图**：调整关联关系

### 4.3 词汇删除流程

提议删除  评估依赖  确认无引用  删除条目  更新日志  更新关系图

#### 步骤说明：
1. **提议删除**：说明删除理由
2. **评估依赖**：检查跨文档引用
3. **确认无引用**：确保没有文章依赖该词汇
4. **删除条目**：从主表移除
5. **更新日志**：记录删除信息
6. **更新关系图**：移除关联关系

## 五、术语定义规范校验

### 5.1 格式校验规则

| 校验项 | 要求 |
|--------|------|
| 词条命名 | 使用「**词条名**」格式，加粗显示 |
| 定义长度 | 一句完整定义，不超过50字 |
| 状态标签 | 必须包含//之一 |
| 模型定位 | 必须从预设分类中选择 |
| 核心锚点 | 必须关联旋量-太极操作 |
| 来源文章 | 必须标注引用来源 |

### 5.2 内容一致性校验

| 校验项 | 要求 |
|--------|------|
| 核心锚点一致 | 同一操作（平方/开方等）定义必须统一 |
| 术语使用一致 | 避免同义词并存 |
| 逻辑链条一致 | 关联词之间必须有明确逻辑关系 |
| 跨章节一致 | 同一词汇在不同章节含义必须相同 |

## 六、跨文档引用一致性检查

### 6.1 检查范围

- `docs/` 目录下所有 Markdown 文件
- `articles/` 目录下所有 HTML 文件
- `scripts/` 目录下注释和文档字符串

### 6.2 检查方法

```python
# 检查流程
1. 提取主表所有词条名
2. 遍历所有文档
3. 标记非标准术语用法
4. 生成不一致报告
```

### 6.3 不一致处理

| 不一致类型 | 处理方式 |
|------------|----------|
| 术语拼写差异 | 统一为标准拼写 |
| 定义冲突 | 组织审查会议解决 |
| 缺失标注 | 补充来源引用 |
| 过时引用 | 更新至最新版本 |

## 七、版本控制与变更记录

### 7.1 变更日志格式

```markdown
日期 | 操作 | 词条 | 详情 | 来源 | 审查者
-----|------|------|------|------|--------
YYYY-MM-DD | 新增/修正/删除 | 词条名 | 变更内容摘要 | 来源文章/对话 | 审查者
```

### 7.2 版本标记规则

| 版本类型 | 标记 | 触发条件 |
|----------|------|----------|
| 重大更新 | vX.0.0 | 核心术语定义变更 |
| 次要更新 | vX.Y.0 | 新增词汇或关联关系调整 |
| 微小更新 | vX.Y.Z | 格式调整或文字修正 |

## 八、定期审核机制

### 8.1 审核周期

| 审核类型 | 周期 | 负责人 |
|----------|------|--------|
| 日常检查 | 每周 | 词汇表维护者 |
| 全面审核 | 每季度 | 核心团队 |
| 版本发布审核 | 发布前 | 审查委员会 |

### 8.2 审核内容

| 审核项 | 检查内容 |
|--------|----------|
| 完整性 | 是否有遗漏的核心术语 |
| 一致性 | 是否存在定义冲突 |
| 更新及时性 | 是否需要根据新文章更新 |
| 状态准确性 | 状态标签是否需要调整 |
| 引用完整性 | 是否所有引用都已标注来源 |

---

## 九、冲突解决规则

### 9.1 冲突类型与解决策略

| 冲突类型 | 解决策略 |
|----------|----------|
| 定义分歧 | 组织专题讨论，达成共识 |
| 优先级冲突 | 以核心模型定义为准 |
| 引用冲突 | 追溯来源，确认最新版本 |
| 命名冲突 | 统一术语命名规范 |

### 9.2 冲突升级流程

发现冲突  初步协调  专题讨论  投票表决  记录决议  更新文档
