# 用户反馈处理流程

> 本文档定义了旋量-太极模型项目（spinor-taiji-model）的用户反馈标准化处理流程，确保所有反馈得到及时、系统、闭环的处理。

## 一、流程总览

```
用户提交反馈 → 自动标记 triage → 筛选分类 → 评估优先级 → 分析确认 → 实施修改 → 部署上线 → 通知反馈者 → 关闭归档
```

## 二、各阶段详细说明

### 阶段 1：收集

| 项目 | 说明 |
|------|------|
| 负责人 | 项目维护者 |
| 频率 | 每周一、三、五检查 |
| 渠道 | GitHub Issues（主）、微信公众号后台留言（辅） |
| 工具 | GitHub Issues 标签系统 |

**操作步骤**：
1. 打开 [GitHub Issues](https://github.com/skyboyhjj/spinor-taiji-model/issues)
2. 筛选 `status:triage` 标签的 Issue
3. 查看微信公众号后台留言（每周汇总一次）
4. 将公众号有效反馈手动创建为 GitHub Issue

### 阶段 2：筛选

| 项目 | 说明 |
|------|------|
| 负责人 | 项目维护者 |
| 时限 | 收到反馈后 24 小时内完成 |

**操作步骤**：
1. 判断反馈是否有效（非垃圾信息、非重复内容）
2. 打上对应标签：
   - 来源标签：`source:statement` / `source:glossary` / `source:other`
   - 类型标签：`type:correction` / `type:suggestion` / `type:question` / `type:appreciation` / `type:bug`
3. 重复反馈标记 `status:duplicate`，在评论中关联原 Issue（`Duplicate of #XX`）
4. 无效反馈标记 `status:wontfix`，附简要说明后关闭
5. 移除 `status:triage`，打上 `status:accepted`

### 阶段 3：优先级评估

| 优先级 | 判定条件 | 响应时限 |
|:---:|------|:---:|
| `priority:critical` | 事实性错误（物理定义、引用错误）、链接全站失效 | 24h |
| `priority:high` | 表述可能导致严重误解、多处关联内容需同步修改 | 72h |
| `priority:medium` | 单一词条优化、表述精炼、新增词条建议 | 1 周 |
| `priority:low` | 排版美化、非核心表述调整、个人偏好 | 2 周 |

### 阶段 4：分析

| 项目 | 说明 |
|------|------|
| 负责人 | 项目维护者 + 领域专家（必要时） |
| 交付物 | Issue 评论中的分析结论 |

**按类型分析**：

- **纠错类** (`type:correction`)：验证事实准确性，确认修改方案。涉及物理定义的需查证权威来源。
- **建议类** (`type:suggestion`)：评估可行性，判断是否与模型核心逻辑一致。重大建议需团队讨论。
- **疑问类** (`type:question`)：准备解答内容，考虑是否需更新文档以预防同类疑问。
- **赞赏类** (`type:appreciation`)：回复感谢，可选择性公开（征得同意后）。

### 阶段 5：响应

| 项目 | 说明 |
|------|------|
| 负责人 | 项目维护者 |
| 首次响应时限 | 48 小时内 |
| 交付物 | Issue 评论回复 |

**回复模板**：

```
感谢你的反馈！[简要确认内容]

[处理结论]

[后续计划/已修改的说明]
```

### 阶段 6：实施

| 项目 | 说明 |
|------|------|
| 负责人 | 项目维护者 |
| 标签 | `status:in-progress` |

**操作步骤**：
1. 创建修复分支：`hotfix/issue-XX` 或 `feature/issue-XX`
2. 修改相关文件
3. 提交时在 commit message 中关联 Issue：`Closes #XX`
4. 推送并创建 Pull Request
5. 合并后 Cloudflare Pages 自动部署

### 阶段 7：跟进

| 项目 | 说明 |
|------|------|
| 负责人 | 项目维护者 |
| 标签 | `status:resolved` |

**操作步骤**：
1. 部署上线后验证修复效果
2. 在 Issue 中回复通知反馈者
3. 标记 `status:resolved` 并关闭 Issue
4. 更新 CHANGELOG.md（如涉及版本变更）

## 三、跨渠道协同

### 微信公众号 → GitHub Issues

1. 每周一汇总上周公众号留言
2. 筛选有效反馈
3. 手动创建 GitHub Issue，标注 `source:other`
4. 在 Issue 中注明"来自微信公众号用户反馈"

### 邮件反馈

1. 如收到邮件反馈，评估内容
2. 手动创建 GitHub Issue
3. 通过邮件回复感谢并引导至 GitHub Issue 跟进

## 四、定期汇总

### 周报（每周一）

- 发布位置：GitHub Discussions → "反馈周报"类别
- 内容：本周新增/已解决/未解决数量、热门反馈主题、本周亮点

### 月报（每月 1 日）

- 发布位置：CHANGELOG.md + 微信公众号推送
- 内容：月度统计、趋势分析、采纳率、版本更新计划

## 五、指标看板

|------|:---:|:---:|

## 六、标签速查

### 来源
- `source:statement` — 声明页面
- `source:glossary` — 词汇表页面
- `source:other` — 其他

### 类型
- `type:correction` — 内容纠错
- `type:suggestion` — 改进建议
- `type:question` — 疑问求助
- `type:appreciation` — 赞赏
- `type:bug` — 技术故障

### 优先级
- `priority:critical` — 阻断
- `priority:high` — 重要
- `priority:medium` — 一般
- `priority:low` — 建议

### 状态
- `status:triage` — 待分类
- `status:accepted` — 已采纳
- `status:in-progress` — 处理中
- `status:resolved` — 已解决
- `status:wontfix` — 不予处理
- `status:duplicate` — 重复
