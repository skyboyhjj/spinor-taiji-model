## 一、代码提交最佳实践总结

### 1.1 提交信息规范

遵循 [Conventional Commits](https://www.conventionalcommits.org/) 规范：

```
<type>(<scope>): <description>

[optional body]

[optional footer]
```

**类型说明**：

| 类型 | 含义 | 示例 |
|------|------|------|
| `feat` | 新增功能 | `feat(feedback): 新增图片上传功能` |
| `fix` | 修复bug | `fix(button): 修复反馈按钮滚动监听失效` |
| `refactor` | 重构代码 | `refactor: 提取反馈功能为共享资源文件` |
| `docs` | 文档更新 | `docs: 更新API文档` |
| `style` | 样式调整 | `style: 优化反馈弹窗样式` |
| `test` | 测试相关 | `test: 添加反馈功能单元测试` |
| `chore` | 构建/工具 | `chore: 更新构建脚本` |
| `perf` | 性能优化 | `perf: 优化图片加载速度` |

### 1.2 提交粒度原则

- **原子化提交**: 每次提交只包含一个逻辑变更
- **避免大提交**: 复杂功能拆分为多个小提交
- **相关变更一起提交**: 功能实现+测试+文档一起提交

### 1.3 分支管理规范

- `main`: 主分支，保持可部署状态
- `feature/*`: 功能开发分支
- `fix/*`: Bug修复分支
- `hotfix/*`: 紧急修复分支

### 2.2 详细验证步骤

#### 步骤1：语法检查（静态代码分析）

| 检查项 | 工具 | 通过标准 |
|--------|------|---------|
| Python语法 | `py_compile` | 无语法错误 |
| JavaScript语法 | 浏览器控制台 | 无SyntaxError |
| HTML结构 | 浏览器解析 | 无标签闭合错误 |

**执行命令**:
```bash
python -c "import py_compile; py_compile.compile('scripts/build.py', doraise=True)"
python -c "import py_compile; py_compile.compile('scripts/convert_vocab.py', doraise=True)"
```

#### 步骤2：构建验证

| 检查项 | 工具 | 通过标准 |
|--------|------|---------|
| 构建脚本 | `scripts/build.py` | 构建成功，无错误 |
| 目录结构 | `ls dist/` | `articles/`、`assets/`、`docs/`、`functions/` 目录存在 |
| 资源文件 | 检查文件存在 | `dist/assets/feedback.css`、`dist/assets/feedback.js` 存在 |

**执行命令**:
```bash
python scripts/build.py
```

#### 步骤3：功能验证

| 检查项 | 验证方式 | 通过标准 |
|--------|---------|---------|
| 反馈按钮显示 | 滚动页面 > 300px | 按钮出现 |
| 反馈按钮隐藏 | 滚动页面回顶部 | 按钮隐藏 |
| 弹窗打开 | 点击反馈按钮 | 弹窗正常显示 |
| 表单验证 | 输入测试内容 | 中文≥5字、英文≥10字符通过 |
| 弹窗关闭 | 点击关闭按钮 | 弹窗正常关闭 |

**执行命令**:
```bash
python scripts/test_interactions.py
```

#### 步骤4：提交信息检查

| 检查项 | 规则 | 通过标准 |
|--------|------|---------|
| 格式 | `type(scope): description` | 符合规范 |
| 类型 | `feat/fix/refactor/docs/style/test/chore/perf` | 类型有效 |
| 长度 | 首行 ≤ 72字符 | 长度合规 |

## 四、异常处理流程

### 4.1 验证失败处理

| 失败类型 | 处理方式 | 责任人 |
|---------|---------|--------|
| 语法错误 | 修复代码后重新验证 | 开发人员 |
| 构建失败 | 检查构建脚本和依赖 | 开发人员 |
| 功能验证失败 | 修复功能后重新验证 | 开发人员 |
| 提交信息格式错误 | 修改提交信息 | 开发人员 |

### 4.2 紧急情况处理

**紧急修复流程**:
1. 创建 `hotfix/*` 分支
2. 跳过部分非关键验证（需团队确认）
3. 提交并推送
4. 后续补充完整验证

**跳过验证命令**:
```bash
git commit --no-verify -m "hotfix: 紧急修复xxx"
```

> **注意**: 跳过验证需在团队群中说明原因

## 七、执行步骤

### 7.1 开发流程

1. 创建功能分支: `git checkout -b feature/xxx`
2. 编写代码
3. 执行本地验证: `python scripts/validate_before_commit.py`
4. 提交代码: `git commit -m "feat(xxx): description"`
5. 推送分支: `git push origin feature/xxx`
6. 创建Pull Request

### 7.2 主分支合并流程

1. 代码审查通过
2. CI/CD验证通过
3. 合并到main分支
4. 触发部署

## 八、附录

### 8.1 常用命令

```bash
# 执行完整验证
python scripts/validate_before_commit.py

# 仅执行构建验证
python scripts/build.py

# 跳过验证强制提交（仅紧急情况）
git commit --no-verify -m "hotfix: xxx"

# 查看提交历史
git log --oneline --graph

# 查看提交详情
git show <commit-hash>
```

### 8.2 提交信息示例

**好的提交**:
```
feat(feedback): 新增图片上传预览功能

- 添加图片拖拽上传支持
- 实现图片预览和删除功能
- 限制上传数量最多3张

**不好的提交**:
```
修改了反馈功能
