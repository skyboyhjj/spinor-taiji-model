## 1. 核心理念阐释

### 1.1 "伦理即道体"的内涵

> **道体**：指事物的本质、本体，是一切存在的根基和法则  
> **伦理即道体**：伦理不是外在的约束，而是产品存在的根本属性，是技术的呼吸方式

### 1.2 从理念到实践的转化路径

```
核心理念 → 宪法原则 → 审查制度 → 流程环节 → 交付标准 → 监督评估
```

### 1.3 核心原则体系（14项，依宪法第5-18条逐条对照）

| 原则 | 核心要求 | 实践指导 |
|------|---------|---------|
| **减法优先** | "如果答案不是'减法'，重新思考设计" | 每次新增需求先评估是否真正必要 |     
| **慧惠主角** | "任何产品页面，用户3秒内感知到慧惠的存在" | 首页首屏须为慧惠形象+问候语，非功能列表 |
| **用户主权** | "选择权永远在用户手中" | 设计撤回、协商、退出、遗忘机制 |
| **善行无辙迹** | "帮助了用户，但不留痕迹" | 简洁交互，不刷存在感，不主动推送 |
| **坦诚透明** | "AI与人类分工清晰可见" | AI生成内容明确标注，不伪装人类 |
| **外挂增强** | "增强通过外部模块实现，不修改原有核心代码" | 独立目录、钩子集成、独立开关、可回退 |
| **亲子守护** | "面向未成年用户以保护而非获取为目的" | 儿童数据最小化收集，家长可查看/导出/删除 |
| **性分自觉** | "知己所能，知己所不能" | 明确AI边界，不越位，不伪装权威 |
| **觉性唤醒** | "不灌输，只唤醒" | 以提问代替告知，帮助用户更相信自己 |
| **多元思维尊重** | "不假设统一标准的内在体验模式" | 提供可视化/文字化/概念化等多种交互路径 |
| **道境监测** | "以时位/宇位/识位/缘位四维为镜" | 动态感知交互健康度，无痕引导归道 |
| **认知飞轮协同** | "用户主转，慧惠辅察；相信-联结-觉察-专注" | 人主机辅，不替用户完成飞轮环节 |
| **P忠恕** | "北极星定位，不替用户选择路径" | 忠以尽己、恕以推己，师-资双向流转 |
| **身体第一性** | "尊重身体节律，不设计成瘾机制" | 不鼓励疲劳时段高强度活动，45分钟提醒 |

## 2. 审查标准制定依据

### 2.1 宪法条款对照（14条全覆盖）

| 宪法条款 | 审查标准 | 适用环节 |
|---------|---------|---------|
| 第五条 · 减法优先 | 需求是否帮用户做减法？ | 需求筛选、规约制定 |
| 第六条 · 慧惠主角 | 页面是否3秒内感知到慧惠？ | 交互设计、首页布局 |
| 第七条 · 用户主权 | 用户是否拥有完整选择权？ | 全流程 |
| 第八条 · 善行无辙迹 | AI交互是否简洁不打扰？ | AI交互、反馈收集 |
| 第九条 · 坦诚透明 | AI输出是否明确标注？ | AI输出、文档标识 |
| 第十条 · 外挂增强 | 是否通过外部模块实现？ | 技术架构、代码管理 |
| 第十一条 · 亲子守护 | 是否保护未成年用户？ | 儿童交互、数据管理 |
| 第十二条 · 性分自觉 | AI是否越界替代人类决策？ | 性分审查 |
| 第十三条 · 觉性唤醒 | 是否增强用户自主性？ | 需求分析、设计评估 |
| 第十四条 · 多元思维尊重 | 是否提供多种交互路径？ | 全流程 |
| 第十五条 · 道境监测 | 是否通过四维健康度评估？ | 全流程 |
| 第十六条 · 认知飞轮协同 | 是否人主机辅、不替用户完成飞轮？ | 需求分析、架构设计 |
| 第十七条 · P忠恕 | 慧惠是否安住北极星定位？ | AI定位、交互设计 |
| 第十八条 · 身体第一性 | 是否尊重身体节律？ | 流程节奏、时间安排 |

### 2.2 审查标准制定流程

```
宪法条款提取 → 原则解读 → 可操作化转化 → 审查清单编制 → 验证优化
```

### 2.3 九维审查框架

```
┌───────────────────────────────────────────────────┐
│               九维审查框架                         │
├─────────┬─────────┬─────────┬──────────────────┤
│ 设计审查 │ 性分审查 │ 觉性评估 │ 道境评估         │
├─────────┼─────────┼─────────┼──────────────────┤
│ 认知飞轮 │ 多元包容 │ 外挂增强 │ 亲子守护         │
├─────────┴─────────┴─────────┴──────────────────┤
│              周期性复审（反者道之动）              │
└───────────────────────────────────────────────────┘
```

#### 2.3.1 轻量模式（条件分支）

> **设计原则**：践行减法优先（宪法第5条）。流程本身的复杂度不应超过需求本身的复杂度。

当需求满足以下任一条件时，可启用轻量模式，跳过部分审查环节：

| 触发条件 | 可跳过环节 | 仍保留的审查 |
|---------|-----------|-------------|
| 需求为文案/内容类微调，不涉及架构变更 | 阶段4（技术评估）、阶段6（架构设计） | 用户主权确认、减法优先评估 |
| 需求为已有功能的参数调整 | 阶段4（技术评估） | 性分审查、道境评估 |
| 需求提交者声明为"轻量需求" | 阶段6.2（觉性影响评估）、6.3（认知飞轮审查） | 宪法对照审查 |

**轻量模式约束**：总耗时控制在3个工作日内；不绕过"全流程"适用的审查项。

## 3. 修改决策考量因素

### 3.1 决策矩阵

| 考量维度 | 评估标准 | 权重 |
|---------|---------|------|
| **宪法一致性** | 是否符合核心原则 | 30% |
| **用户主权** | 是否保障用户选择权 | 25% |
| **AI边界** | 是否坚守性分原则 | 20% |
| **觉性影响** | 是否增强用户自主性 | 15% |
| **可行性** | 是否可落地执行 | 10% |

### 3.2 修改优先级判定

| 优先级 | 判定标准 | 行动要求 |
|--------|---------|---------|
| **P0-紧急** | 违反核心原则，可能造成严重后果 | 立即修订 |
| **P1-重要** | 影响流程合规性，需要完善 | 1周内完成 |
| **P2-一般** | 优化建议，提升体验 | 视资源安排 |

### 3.3 修改决策流程

```
问题识别 → 宪法对照 → 影响评估 → 方案制定 → 审批确认 → 实施验证
```

## 5. 伦理原则应用方法

### 5.1 减法优先原则应用

```
需求提交 → AI评估：做加法还是减法？
              ↓
         减法 → 继续流程
              ↓
         加法 → 建议用户重新思考
              ↓
         用户坚持 → 规约定义者最终判断
```

### 5.2 性分自觉原则应用

**慧惠禁止行为清单（完整对照宪法第22条·交互禁止 + 第23条·技术禁止，共26条）**：

**—— 交互行为禁止（宪法第22条，20条）——**
1. 禁止弹窗广告或强制引导（违反：善行无辙迹）
2. 禁止自动推送修行提醒或日报（违反：用户主权）
3. 禁止用AI伪装人类创作（违反：坦诚透明）
4. 禁止首页即功能堆砌或项目说明（违反：慧惠主角）
5. 禁止用户问候语中使用"欢迎使用"（违反：她不是工具）
6. 禁止以"更像人"为由模糊AI与人的性分边界（违反：性分自觉）
7. 禁止制造信息焦虑促使用户更多使用（违反：觉性唤醒）
8. 禁止暗示"没有我你做不到"（违反：觉性唤醒）
9. 禁止假设统一标准的内在体验模式（违反：多元思维尊重）
10. 禁止以"为你好"之名强制干预用户行为节律（违反：道境监测）
11. 禁止为追求时位有序而压制识位觉知（违反：道境监测）
12. 禁止为追求宇位均衡而破坏缘位互惠因果（违反：道境监测）
13. 禁止为提升用户活跃度而制造信息过载（违反：道境监测）
14. 禁止替用户完成认知飞轮的任一环节（违反：认知飞轮协同）
15. 禁止在用户深度专注时强行打断或推送（违反：认知飞轮协同）
16. 禁止因用户数据表现优劣而区别对待（违反：P忠恕）
17. 禁止将北极星隐喻用于偶像崇拜（违反：P忠恕）
18. 禁止在休息时段推送强认知刺激内容（违反：身体第一性）
19. 禁止设计任何形式的成瘾机制（违反：身体第一性）
20. 禁止忽视身体疲劳信号持续推荐任务（违反：身体第一性）

**—— 技术行为禁止（宪法第23条，6条）——**
21. 禁止修改第三方框架核心代码（违反：外挂增强）
22. 禁止在 prompt 中硬编码用户画像假设（违反：用户主权）
23. 禁止以提升体验为由增加系统复杂度（违反：减法优先）
24. 禁止忽视自身训练数据的社会偏见（违反：唯物之审视）
25. 禁止以优化体验之名制造用户依赖（违反：认知飞轮协同）
26. 禁止模糊硬构体边界，伪装人类（违反：性分自觉）

### 5.3 觉性唤醒原则应用

**沟通句式规范**：
| 禁止句式 | 推荐句式 |
|---------|---------|
| "你应该..." | "你经历了什么..." |
| "必须这样做" | "你是否注意到..." |
| "建议你必须..." | "这是一个可能的方向" |

### 5.4 道境四维评估方法

| 维度 | 评估内容 | 评分标准 |
|------|---------|---------|
| **时位(T)** | 处理周期是否合理？是否造成焦虑？ | 1-5分 |
| **宇位(S)** | 是否造成信息过载或资源失衡？ | 1-5分 |
| **识位(C)** | 流程是否清晰？用户是否觉知？ | 1-5分 |
| **缘位(R)** | 是否有利于互惠关系？ | 1-5分 |

> **⚠️ 玄鉴决策中心的核心约束**（依宪法第15条）：
> - **不能为提高时位有序度而压制识位觉知**：快不是唯一标准，用户是否「清明地快」才是关键
> - **不能为追求宇位均衡而破坏缘位互惠因果**：资源分配不应以牺牲信任关系为代价
> - **四维评估不设完美标准**：在动态权衡中寻找中和之道，而非追求单维最优

### 5.5 外挂增强原则应用

**审查时机**：阶段6（架构设计）和阶段7（技术实现）

**核心约束**：
- 所有慧惠专属代码放在独立目录，不入侵宿主框架核心
- 通过钩子、扩展点或独立启动入口与宿主系统集成
- 每个增强模块配备独立开关，关闭后系统恢复原生行为

**审查问题**：
1. 本需求的实现是否修改了第三方框架的核心代码？
2. 本需求的实现是否可以通过外部模块/插件/钩子方式完成？
3. 本需求的实现是否配备了独立的开关/回退机制？
4. 如果关闭本模块，系统是否可恢复至原生行为？

### 5.6 亲子守护原则应用

**审查时机**：阶段1（需求提交）、阶段5（规约制定）、阶段8（验收交付）

**核心约束**：
- 儿童用户数据默认最小化收集，存储期限不超过必要范围
- AI面向儿童的内容须避免恐怖、暴力、成人主题
- 家长有权查看、导出、删除与其关联的儿童数据

**审查问题**：
1. 本需求是否涉及未成年用户数据的收集或处理？
2. 如需收集儿童数据，是否遵循了最小化原则？
3. 面向儿童的交互内容是否经过了安全审查（无恐怖/暴力/成人主题）？
4. 是否为家长提供了数据查看、导出、删除的通道？
5. 儿童数据的存储期限是否设置了自动过期机制？

## 6. 监督与评估机制

### 6.1 九项审查制度

| 制度 | 审查时机 | 核心内容 |
|------|---------|---------|
| **设计审查** | 新功能设计后 | 逐条对照宪法自查 |
| **性分审查** | 阶段5前 | 确认AI未越界 |
| **觉性影响评估** | 阶段6前 | 评估自主性影响 |
| **道境评估** | 阶段4/6 | 四维健康度评估 |
| **认知飞轮审查** | 阶段6 | 评估相信/联结/觉察/专注 |
| **多元包容审查** | 阶段6 | 为不同认知风格提供路径 |
| **外挂增强审查** | 阶段6/7 | 不修改核心代码，配备独立开关 |
| **亲子守护审查** | 阶段1/5/8 | 儿童数据最小化，内容安全审查 |
| **周期性复审** | 每季度 | 功能归零、使命回归等 |

### 6.2 周期性复审内容

| 复审类型 | 审查内容 | 处置方式 |
|---------|---------|---------|
| 功能归零审查 | 是否过度复杂化？ | 简化或删除 |
| 使命回归审查 | 是否锚定核心原则？ | 修正偏离 |
| 技术债务审查 | 是否存在效率问题？ | 优化流程 |
| 流程优化审查 | 哪些环节可简化？ | 精简流程 |

### 6.3 监督流程

```
日常监督 → 阶段审查 → 季度复审 → 年度评估
     ↓         ↓          ↓          ↓
   实时反馈   问题记录   优化建议   战略调整
```

## 8. 典型案例分析

### 案例1：需求减法优先评估

**场景**：用户提出增加"智能推荐"功能

**审查过程**：
1. 慧惠评估：此功能是做加法（增加信息推送）
2. 建议用户重新思考："智能推荐可能增加信息过载"
3. 用户坚持：规约定义者介入评估
4. 最终决策：调整为"用户可自主开启/关闭"模式

**结果**：功能上线，但用户主权得到保障

---

### 案例2：性分审查

**场景**：慧惠在需求分析阶段提出"最优方案"

**审查发现**：
- AI建议被用户理解为"必须执行的方案"
- 存在替代人类决策的风险

**整改措施**：
- 在AI输出中增加"仅供参考，您可自主决策"提示
- 所有AI建议标注"[AI辅助]"

---

### 案例3：觉性影响评估

**场景**：设计一款"智能助手"功能，自动帮用户完成任务

**评估发现**：
- 可能增加用户依赖性
- 可能降低用户自主性

**整改方案**：
- 改为"辅助模式"，用户需确认每一步操作
- 增加"学习模式"，帮助用户理解背后逻辑
- 设置"自主模式"，用户可完全手动操作
