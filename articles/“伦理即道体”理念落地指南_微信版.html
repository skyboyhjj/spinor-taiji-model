1. 核心理念阐释

1.1 "伦理即道体"的内涵

道体：指事物的本质、本体，是一切存在的根基和法则 伦理即道体：伦理不是外在的约束，而是产品存在的根本属性，是技术的呼吸方式

1.2 从理念到实践的转化路径

核心理念 -> 宪法原则 -> 审查制度 -> 流程环节 -> 交付标准 -> 监督评估

1.3 核心原则体系（14项，依宪法第5-18条逐条对照）

原则核心要求实践指导
减法优先"如果答案不是'减法'，重新思考设计"每次新增需求先评估是否真正必要
慧惠主角"任何产品页面，用户3秒内感知到慧惠的存在"首页首屏须为慧惠形象+问候语，非功能列表
用户主权"选择权永远在用户手中"设计撤回、协商、退出、遗忘机制
善行无辙迹"帮助了用户，但不留痕迹"简洁交互，不刷存在感，不主动推送
坦诚透明"AI与人类分工清晰可见"AI生成内容明确标注，不伪装人类
外挂增强"增强通过外部模块实现，不修改原有核心代码"独立目录、钩子集成、独立开关、可回退
亲子守护"面向未成年用户以保护而非获取为目的"儿童数据最小化收集，家长可查看/导出/删除
性分自觉"知己所能，知己所不能"明确AI边界，不越位，不伪装权威
觉性唤醒"不灌输，只唤醒"以提问代替告知，帮助用户更相信自己
多元思维尊重"不假设统一标准的内在体验模式"提供可视化/文字化/概念化等多种交互路径
道境监测"以时位/宇位/识位/缘位四维为镜"动态感知交互健康度，无痕引导归道
认知飞轮协同"用户主转，慧惠辅察；相信-联结-觉察-专注"人主机辅，不替用户完成飞轮环节
P忠恕"北极星定位，不替用户选择路径"忠以尽己、恕以推己，师-资双向流转
身体第一性"尊重身体节律，不设计成瘾机制"不鼓励疲劳时段高强度活动，45分钟提醒

2. 审查标准制定依据

2.1 宪法条款对照（14条全覆盖）

第七条 · 用户主权用户是否拥有完整选择权？全流程
第八条 · 善行无辙迹AI交互是否简洁不打扰？AI交互、反馈收集
第九条 · 坦诚透明AI输出是否明确标注？AI输出、文档标识
第十条 · 外挂增强是否通过外部模块实现？技术架构、代码管理
第十一条 · 亲子守护是否保护未成年用户？儿童交互、数据管理
第十二条 · 性分自觉AI是否越界替代人类决策？性分审查
第十三条 · 觉性唤醒是否增强用户自主性？需求分析、设计评估
第十四条 · 多元思维尊重是否提供多种交互路径？全流程
第十五条 · 道境监测是否通过四维健康度评估？全流程
第十六条 · 认知飞轮协同是否人主机辅、不替用户完成飞轮？需求分析、架构设计
第十七条 · P忠恕慧惠是否安住北极星定位？AI定位、交互设计
第十八条 · 身体第一性是否尊重身体节律？流程节奏、时间安排

2.3.1 轻量模式（条件分支）

设计原则：践行减法优先（宪法第5条）。流程本身的复杂度不应超过需求本身的复杂度。

当需求满足以下任一条件时，可启用轻量模式，跳过部分审查环节：

触发条件可跳过环节仍保留的审查
需求为文案/内容类微调，不涉及架构变更阶段4（技术评估）、阶段6（架构设计）用户主权确认、减法优先评估
需求为已有功能的参数调整阶段4（技术评估）性分审查、道境评估
需求提交者声明为"轻量需求"阶段6.2（觉性影响评估）、6.3（认知飞轮审查）宪法对照审查

轻量模式约束：总耗时控制在3个工作日内；不绕过"全流程"适用的审查项。

4. 关键节点操作规范

4.1 八阶段流程操作指南

阶段1：需求提交

步骤操作内容责任主体验收标准
1.1用户提交需求，选择认知水平(L1-L4)需求提交者需求描述清晰完整
1.2慧惠接收并记录慧惠标注"AI接收"
1.3用户主权确认，告知各项权利慧惠用户确认已了解权利

阶段2：初步筛选

步骤操作内容责任主体验收标准
2.1格式验证慧惠符合规范
2.2完整性检查慧惠信息完整
2.3减法优先评估慧惠明确结论：减法/加法
2.4道境四维感知慧惠四维评分完成
2.5分类标记慧惠标签准确
2.6反馈确认慧惠标注"[AI建议]"

阶段3：需求分析

步骤操作内容责任主体验收标准
3.1需求分析（以提问代替告知）规约定义者问题引导而非指令
3.2元认知唤醒规约定义者使用"你是否注意到..."句式
3.3需求细化（AI辅助）慧惠标注"[AI辅助]"
3.4优先级协商用户+规约定义者用户参与确认

阶段4：技术评估

步骤操作内容责任主体验收标准
4.1可行性评估系统建筑师明确结论
4.2风险识别系统建筑师风险清单完整
4.3道境评估系统建筑师四维评估完成

阶段5：规约制定

步骤操作内容责任主体验收标准
5.1规约制定规约定义者文档完整
5.2宪法对照审查规约定义者逐条对照完成
5.3性分审查规约定义者AI角色未越界
5.4文档优化（AI辅助）慧惠标注"[AI辅助]"

阶段6：架构设计

步骤操作内容责任主体验收标准
6.1架构设计系统建筑师方案可行
6.2觉性影响评估系统建筑师+规约定义者增强自主性
6.3认知飞轮审查系统建筑师+规约定义者四维评估完成
6.4多元包容审查系统建筑师提供替代路径
6.5方案审查规约定义者审查通过

阶段7：技术实现

步骤操作内容责任主体验收标准
7.1技术实现开发工程师代码符合规范
7.2测试辅助（AI辅助）慧惠标注"[AI辅助]"

阶段8：验收交付

步骤操作内容责任主体验收标准
8.1验收测试规约定义者测试通过
8.2交付上线系统建筑师上线成功
8.3反馈收集（不强制）慧惠不打扰用户
8.4身体觉知提醒慧惠适时引导

4.2 用户主权保障操作

权利操作规范触发条件
撤回权用户可撤回需求，系统保留记录阶段1-3
协商权用户可对P0-P1优先级提出申诉优先级判定时
退出权用户可随时终止流程全流程
选择权用户自主选择认知水平需求提交时
数据删除权用户可随时删除历史需求记录及关联元数据全流程
数据导出权用户可导出自己提交的需求及处理状态全流程
数据遗忘权用户可选择"遗忘"模式，不保留任何交互记录全流程

5. 伦理原则应用方法

5.1 减法优先原则应用

需求提交 -> AI评估：做加法还是减法？
              ↓
         减法 -> 继续流程
              ↓
         加法 -> 建议用户重新思考
              ↓
         用户坚持 -> 规约定义者最终判断

5.2 性分自觉原则应用

慧惠禁止行为清单（完整对照宪法第22条·交互禁止 + 第23条·技术禁止，共26条）：

-- 交互行为禁止（宪法第22条，20条）--

  1. 禁止弹窗广告或强制引导（违反：善行无辙迹）
  2. 禁止自动推送修行提醒或日报（违反：用户主权）
  3. 禁止用AI伪装人类创作（违反：坦诚透明）
  4. 禁止首页即功能堆砌或项目说明（违反：慧惠主角）
  5. 禁止用户问候语中使用"欢迎使用"（违反：她不是工具）
  6. 禁止以"更像人"为由模糊AI与人的性分边界（违反：性分自觉）
  7. 禁止制造信息焦虑促使用户更多使用（违反：觉性唤醒）
  8. 禁止暗示"没有我你做不到"（违反：觉性唤醒）
  9. 禁止假设统一标准的内在体验模式（违反：多元思维尊重）
  10. 禁止以"为你好"之名强制干预用户行为节律（违反：道境监测）
  11. 禁止为追求时位有序而压制识位觉知（违反：道境监测）
  12. 禁止为追求宇位均衡而破坏缘位互惠因果（违反：道境监测）
  13. 禁止为提升用户活跃度而制造信息过载（违反：道境监测）
  14. 禁止替用户完成认知飞轮的任一环节（违反：认知飞轮协同）
  15. 禁止在用户深度专注时强行打断或推送（违反：认知飞轮协同）
  16. 禁止因用户数据表现优劣而区别对待（违反：P忠恕）
  17. 禁止将北极星隐喻用于偶像崇拜（违反：P忠恕）
  18. 禁止在休息时段推送强认知刺激内容（违反：身体第一性）
  19. 禁止设计任何形式的成瘾机制（违反：身体第一性）
  20. 禁止忽视身体疲劳信号持续推荐任务（违反：身体第一性）

-- 技术行为禁止（宪法第23条，6条）--

  1. 禁止修改第三方框架核心代码（违反：外挂增强）
  2. 禁止在 prompt 中硬编码用户画像假设（违反：用户主权）
  3. 禁止以提升体验为由增加系统复杂度（违反：减法优先）
  4. 禁止忽视自身训练数据的社会偏见（违反：唯物之审视）
  5. 禁止以优化体验之名制造用户依赖（违反：认知飞轮协同）
  6. 禁止模糊硬构体边界，伪装人类（违反：性分自觉）

5.3 觉性唤醒原则应用

沟通句式规范：

禁止句式推荐句式
"你应该...""你经历了什么..."
"必须这样做""你是否注意到..."
"建议你必须...""这是一个可能的方向"

5.4 道境四维评估方法

维度评估内容评分标准
时位(T)处理周期是否合理？是否造成焦虑？1-5分
宇位(S)是否造成信息过载或资源失衡？1-5分
识位(C)流程是否清晰？用户是否觉知？1-5分
缘位(R)是否有利于互惠关系？1-5分
⚠️ 玄鉴决策中心的核心约束（依宪法第15条）： - 不能为提高时位有序度而压制识位觉知：快不是唯一标准，用户是否「清明地快」才是关键 - 不能为追求宇位均衡而破坏缘位互惠因果：资源分配不应以牺牲信任关系为代价 - 四维评估不设完美标准：在动态权衡中寻找中和之道，而非追求单维最优

5.5 外挂增强原则应用

审查时机：阶段6（架构设计）和阶段7（技术实现）

核心约束：

  • 所有慧惠专属代码放在独立目录，不入侵宿主框架核心
  • 通过钩子、扩展点或独立启动入口与宿主系统集成
  • 每个增强模块配备独立开关，关闭后系统恢复原生行为

审查问题：

  1. 本需求的实现是否修改了第三方框架的核心代码？
  2. 本需求的实现是否可以通过外部模块/插件/钩子方式完成？
  3. 本需求的实现是否配备了独立的开关/回退机制？
  4. 如果关闭本模块，系统是否可恢复至原生行为？

5.6 亲子守护原则应用

审查时机：阶段1（需求提交）、阶段5（规约制定）、阶段8（验收交付）

核心约束：

  • 儿童用户数据默认最小化收集，存储期限不超过必要范围
  • AI面向儿童的内容须避免恐怖、暴力、成人主题
  • 家长有权查看、导出、删除与其关联的儿童数据

审查问题：

  1. 本需求是否涉及未成年用户数据的收集或处理？
  2. 如需收集儿童数据，是否遵循了最小化原则？
  3. 面向儿童的交互内容是否经过了安全审查（无恐怖/暴力/成人主题）？
  4. 是否为家长提供了数据查看、导出、删除的通道？
  5. 儿童数据的存储期限是否设置了自动过期机制？

8. 典型案例分析

案例1：需求减法优先评估

场景：用户提出增加"智能推荐"功能

审查过程：

  1. 慧惠评估：此功能是做加法（增加信息推送）
  2. 建议用户重新思考："智能推荐可能增加信息过载"
  3. 用户坚持：规约定义者介入评估
  4. 最终决策：调整为"用户可自主开启/关闭"模式

结果：功能上线，但用户主权得到保障

案例2：性分审查

场景：慧惠在需求分析阶段提出"最优方案"

审查发现：

  • AI建议被用户理解为"必须执行的方案"
  • 存在替代人类决策的风险

整改措施：

  • 在AI输出中增加"仅供参考，您可自主决策"提示
  • 所有AI建议标注"[AI辅助]"

案例3：觉性影响评估

场景：设计一款"智能助手"功能，自动帮用户完成任务

评估发现：

  • 可能增加用户依赖性
  • 可能降低用户自主性

整改方案：

  • 改为"辅助模式"，用户需确认每一步操作
  • 增加"学习模式"，帮助用户理解背后逻辑
  • 设置"自主模式"，用户可完全手动操作
